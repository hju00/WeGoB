# [pro] 서비스 센터 문제 풀이

## 1. 문제 개요

로봇이 자동으로 수리 요청을 처리하는 서비스 센터를 운영하고 있다.<br><br>
각 수리 요청은 등급과 업무량 정보를 가지며, 접수시 할당 가능한 로봇이 없으면 대기한다.<br>
지정된 지연 기준 시간을 초과한 수리 요청은 "지연 상태"로 전환된다.<br><br>
각 로봇은 단위 시간당 처리량 정보를 가지며, 할당받을 수 있는 수리 요청이 없으면 대기한다.<br>
★ 업무가 끝난 로봇은 대기 상태인 다른 수리 요청이 존재하면 즉시 새 수리 요청을 할당받는다.<br><br>
수리 요청 접수 및 로봇의 업무 할당은 다음과 같은 우선순위에 의해 진행된다.<br>

- 요청 우선순위  
1) 지연 상태 요청이 최우선  
2) 지연 상태끼리는 접수 시간이 빠른 순  
3) 일반 요청은 등급 높은 순, 같은 등급이면 접수 시간이 빠른 순

- 로봇 우선순위  
1) 처리량 높은 순  
2) 처리량이 같으면 ID 작은 순

- 서비스 센터의 업무 처리 과정은 반드시 다음 순서를 반영해야 한다.  
1) 지연 상태 전환
2) 업무 완료 처리
3) 대기 로봇 재할당
4) 현재 API(접수/추가/제거/평가) 수행

상기한 조건에 따라 수리 요청 접수, 로봇의 업무 할당 및 등급별 업무 평가 과정을 자동화하는 프로그램을 구현해야 한다.

<br>

## 2. 쿼리 분석

### 1. `init(N, M)`
- 초기 1회 호출
- 지연 기준 시간 `N`: $10 \le N \le 100$
- 최대 등급 `M`: $3 \le M \le 10$

-> 수리 요청 접수 대기열을 등급별로 관리할 필요가 있음<br>
-> 높은 등급의 대기 중인 수리 요청이 하나라도 존재하면 그 아래 등급의 수리 요청들은 우선순위가 밀리기 때문<br>
-> 그렇다면 지연 상태인 수리 요청은?<br>

### 2. `receive(mTime, mId, mWorkload, mGrade)`
- 최대 50,000회 호출
- 현재 시각 `mTime`: $1 \le mTime \le 1,000,000$
- 수리 요청 ID `mId`: $1 \le mId \le 50,000$
    - `mId`는 초기 메서드 호출시 1부터 시작, 호출시마다 1씩 증가하는 값이 주어짐
- 수리 요청 업무량 `mWorkload`: $10 \le mWorkload \le 1,000$
- 수리 요청 등급 `mGrade`: $1 \le mGrade \le M$
- 현재 시각(`mTime`)까지의 시뮬레이션 동기화
- 신규 수리 요청 접수
    - 대기 중인 로봇이 존재하면 최고 우선순위 로봇에게 업무 할당
    - 대기 중인 로봇이 없으면 등급별 대기열에 추가

-> 최고 우선순위를 가진 로봇만 요구하는 상태<br>
-> 로봇은 상기한 두 정렬 조건(1: 업무 처리량, 2: ID)을 따르는 우선순위 큐가 필요함<br>

### 3. `add(mTime, rId, mThroughput)`
- 최대 1,000회 호출
- 현재 시각 `mTime`: $1 \le mTime \le 1,000,000$
- 로봇 ID `rId`: $1 \le mId \le 1,000$
    - `rId`는 초기 메서드 호출시 1부터 시작, 호출시마다 1씩 증가하는 값이 주어짐
- 로봇 업무 처리량 `mThroughput`: $5 \le mThroughput \le 20$
- 현재 시각(`mTime`)까지의 시뮬레이션 동기화
- 신규 로봇 추가
    - 할당 대기중인 수리 요청이 존재하면 최고 우선순위 수리 요청을 해당 로봇에게 할당
    - 할당 대기중인 수리 요청이 없으면 로봇 대기열에 추가

-> 최고 우선순위를 가진 수리 요청만 요구하는 상태<br>
-> 수리 요청을 등급별로 나눈다고 할 때, 다음 정렬 조건인 접수 시간 순서를 따르는 우선순위 큐가 등급별로 필요함<br>
-> 지금까지의 논리로는 지연 상태인 수리 요청에 더 높은 우선순위를 부여할 수 없음<br>

### 4. `remove(mTime, rId)`
- 최대 500회 호출
- 현재 시각 `mTime`: $1 \le mTime \le 1,000,000$
- 로봇 ID `rId`: $1 \le mId \le 1,000$
- 현재 시각(`mTime`)까지의 시뮬레이션 동기화
- 지정된 로봇을 제거
    - 대기 리스트 또는 업무 처리중 리스트에 존재하는 상태이므로 지연 삭제 적용
    - 처리 중인 수리 요청이 있으면 할당 해제 및 "수리 진행 현황 초기화"
    - 할당 해제된 수리 요청의 경우
        - 대기 중인 다른 로봇이 있으면 최고 우선순위 로봇에게 업무 할당
        - 대기 중인 다른 로봇이 없으면 다시 등급별 대기열에 추가
        - 지연 상태인 경우 전환 요구됨
- `return`
    - 제거된 로봇이 진행중이었던 수리 요청의 ID 반환
    - 할당된 수리 요청이 없었다면 -1 반환

-> 로봇 대기열이 우선순위 큐이므로 원하는 로봇을 즉시 반환할 수 없어 지연 삭제를 적용해야 함<br>
-> 처리 중인 업무 리스트에 대해서도 즉시 할당 해제 및 수리 요청 ID 반환을 위해 로봇 객체가 할당받은 수리 요청 ID를 저장해야 할 필요가 있음<br>

### 5. `evaluate(mTime, mGrade)`
- 최대 10,000회 호출
- 현재 시각(`mTime`)까지의 시뮬레이션 동기화
- 지정 등급에 해당하는 "처리 완료된 수리 요청"들의 처리 시간(완료 시각 - 접수 시각) 합계를 반환

-> "처리 완료된 수리 요청"에 대해서는 해당 메서드에서만 요구되는 정보임<br>
-> 수리 요청 객체의 총 업무량과, 로봇의 단위 시간당 업무 처리량 정보를 가지고 있으면 충분히 완료 예정 시각을 계산할 수 있음<br>
-> 따라서 "처리 완료된 수리 요청" 객체는 더 이상 사용하지 않고 등급별로 누적 시간 합계만 저장<br>

<br>

## 자료구조 및 알고리즘 설계

### 1. 요청 큐 분리
- `PriorityQueue<Request>[M+1] requestQueue` 사용
- 각 큐는 `received` 오름차순
- 지연 요청은 항상 최상위 등급보다 높은 우선순위를 가진다
    - 이 말을 "최상위 등급보다 한 단계 더 높은 등급"으로 해석하면 등급별 대기열을 구성할 수 있음
    - **등급이 0 ~ M-1까지 존재한다고 할 때, "지연 상태"를 "M" 등급으로 지정**
    - 지연 상태가 된 수리 요청은 `requestQueue[M]`으로 이동시켜 항상 일반 요청보다 먼저 선택되게 구성

### 2. 로봇 큐
- `PriorityQueue<Robot>`: 처리량 내림차순, ID 오름차순
- 빈 로봇 중 최우선 로봇을 즉시 꺼내 할당

### 3. 진행 중 요청 큐
- `inProgress`: `end`(완료 예정 시각) 오름차순
- 현재 시각에 완료되는 요청들을 빠르게 일괄 처리

### 4. 지연 삭제(Lazy Deletion)
- 로봇 제거 시 `removed=true`만 설정하고 큐에서 즉시 탐색 삭제하지 않음
- 큐 top에서 꺼낼 때 제거된 로봇이면 버림
- `remove`로 중단된 기존 요청 객체는 `allocated=-1`로 무효화하고,
  동일 ID의 새 요청 객체를 만들어 재등록하여 처리 충돌을 방지

### 5. 이벤트 기반 시뮬레이션 `repair(time)`
`repair`가 이 풀이의 중심이다.<br>
모든 쿼리 메서드가 호출 즉시 `repair` 메서드를 사용해야 한다.<br>

- `inProgress`(완료 시각 오름차순 PQ)의 다음 완료 시각을 기준으로 프로세스를 진행
- 이벤트 발생 시각(`finishedTime`)마다
  1) 지연 전환
  2) 해당 시각 완료 요청 일괄 처리
  3) 빈 로봇에 요청 재할당
- 다음 완료 이벤트가 `time`보다 뒤면 마지막 이벤트 시각을 `time`으로 두고 프로세스를 진행

즉, **이벤트 완료 시점마다 점프**하며 해당 시간대에 대해 각각 프로세스를 처리한다.

<br>

## 구현 핵심 포인트

- 요청 처리 시간 계산:
  - `end = start + ceil(workload / throughput)`
- 지연 전환은 매 이벤트 시각에 수행:
  - `received + N <= currentEventTime`이면 지연 큐로 이동
- 완료 처리 후에만 빈 로봇 자동 재할당(문제의 단계 순서 반영)
- `remove`로 중단된 요청은 **처음부터 다시 처리**되므로 새 객체로 재등록
- 등급 인덱스는 내부적으로 0-based(`grade-1`)로 관리
